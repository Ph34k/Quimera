<task> 
Implementar a funcionalidade de "Configuração de Hotspots Digitais" no API Backend. Isso 
inclui o modelo de dados, o repositório para persistência no PostgreSQL e os endpoints REST 
para criação (Command) e listagem (Query) de hotspots. O objetivo é suportar a Fase 1 
(Configuração) e Fase 2 (Prospecção) do pipeline. 
</task> 
 
<requirements> 
<functional> 
- Funcionalidade exata que deve implementar: 
  1.  Modelo de dados (`Hotspot`) com `id` (UUID), `url` (string), `plataforma` (enum), `ativo` 
(boolean), `data_criacao` (datetime). 
  2.  Endpoint POST `/api/v1/hotspots` para **criar** um novo hotspot (Command). 
  3.  Endpoint GET `/api/v1/hotspots` para **listar** todos os hotspots ativos (Query). 
  4.  A URL deve ser única e validada como URL válida. 
- Inputs esperados e outputs: 
  * **POST Input**: JSON com `url` (string, max 255), `plataforma` (string: 'INSTAGRAM' ou 
'FACEBOOK'), `ativo` (boolean, opcional, default True). 
  * **POST Output**: Objeto `Hotspot` criado com status 201. 
  * **GET Output**: Lista paginada (implícito) de objetos `Hotspot` com status 200. 
- Comportamento em diferentes cenários: 
  * Deve falhar com status 400 se a URL for inválida ou a plataforma for desconhecida. 
  * Deve falhar com status 409 (Conflict) se a URL já existir (unicidade). 
  * A listagem (GET) deve retornar apenas hotspots com `ativo=True` por padrão.

# 4. infrastructure/hotspot/sqlalchemy_repository.py (Implementação PostgreSQL) 
# ---------------------------------------------------------------------------------- 
 
from sqlalchemy.ext.asyncio import AsyncSession 
from sqlalchemy.future import select 
from sqlalchemy.exc import IntegrityError 
from typing import Optional, List 
 
# Imports de módulos internos 
# Assumindo que Hotspot, HotspotCreateSchema, HotspotRepository, 
HotspotAlreadyExistsError 
# já foram importados ou estão no mesmo módulo em um projeto real. 
# Para este output, assumiremos que as classes acima estão acessíveis. 
 
class HotspotSQLAlchemyRepository(HotspotRepository): 
    """ 
    Implementação do Repositório Hotspot usando SQLAlchemy e AsyncSession. 
 
    Isto mapeia o domínio para a tecnologia de persistência, encapsulando 
    [cite_start]a lógica de acesso a dados[cite: 242]. 
    """ 
    def __init__(self, session: AsyncSession): 
        self._session = session 
 
    async def create(self, hotspot_data: HotspotCreateSchema) -> Hotspot: 
        """ 
        Cria um novo Hotspot no PostgreSQL. 
        """ 
        hotspot_dict = hotspot_data.dict(exclude_unset=True) 
        # O modelo ORM deve aceitar os dados diretamente do schema/dict 
        hotspot = Hotspot(**hotspot_dict) 
        self._session.add(hotspot) 
        try: 
            await self._session.commit() 
            await self._session.refresh(hotspot) 
            return hotspot 
        except IntegrityError as e: 
            # Captura a falha de constraint UNIQUE 



            await self._session.rollback() 
            # Esta verificação é mais robusta que apenas checar antes de inserir 
            if "hotspots_url_key" in str(e): 
                raise HotspotAlreadyExistsError(url=hotspot_data.url) 
            raise e 
 
    async def get_active_hotspots(self, limit: int = 100, offset: int = 0) -> List[Hotspot]: 
        """ 
        Retorna uma lista paginada de hotspots que estão ativos. 
        """ 
        stmt = ( 
            select(Hotspot) 
            .where(Hotspot.ativo == True) 
            .limit(limit) 
            .offset(offset) 
            .order_by(Hotspot.data_criacao.desc()) 
        ) 
        result = await self._session.execute(stmt) 
        # O result.scalars() retorna os objetos Hotspot diretamente 
        return result.scalars().all() 
 
    async def get_by_url(self, url: str) -> Optional[Hotspot]: 
        """Busca um hotspot pela URL.""" 
        stmt = select(Hotspot).where(Hotspot.url == url) 
        result = await self._session.execute(stmt) 
        return result.scalars().first()

# Imports internos: Hotspot, HotspotCreateSchema, HotspotRepository,

class HotspotService: 
    """ 
    Serviço de Aplicação para gerenciar a lógica de negócio dos Hotspots. 
 
    [cite_start]Faz a orquestração entre a API e a camada de Repositório[cite: 179]. 
    """ 
    def __init__(self, repository: HotspotRepository): 
        self._repository = repository 



 
    async def create_hotspot(self, hotspot_data: HotspotCreateSchema) -> 
HotspotResponseSchema: 
        """ 
        Processa a criação de um Hotspot. 
 
        Args: 
            hotspot_data: Dados do novo hotspot. 
 
        Returns: 
            O Hotspot criado. 
 
        Raises: 
            HotspotAlreadyExistsError: Se a URL já estiver cadastrada. 
        """ 
        # A verificação de unicidade é feita no DB (IntegrityError) para evitar race condition, 
        # mas o service chama o repositório que encapsula esse erro de DB 
        new_hotspot = await self._repository.create(hotspot_data) 
        return HotspotResponseSchema.from_orm(new_hotspot) 
 
    async def get_active_hotspots(self, limit: int = 100, offset: int = 0) -> 
List[HotspotResponseSchema]: 
        """ 
        Obtém a lista de Hotspots ativos do Repositório. 
         
        [cite_start]A URL é a principal chave de busca para o Agente Batedor[cite: 45]. 
 
        Args: 
            limit: Número máximo de resultados. 
            offset: Deslocamento para paginação. 
 
        Returns: 
            Lista de Hotspots ativos. 
        """ 
        hotspots = await self._repository.get_active_hotspots(limit=limit, offset=offset) 
        # Mapeia a entidade de domínio para o schema de resposta 
        return [HotspotResponseSchema.from_orm(h) for h in hotspots]

<design_notes> 
1.  **Segregação de Responsabilidade (Clean/DDD/CQRS)**: O design segue o princípio de 
segregação. O `HotspotService` contém a lógica de negócio (embora simples neste caso) e 



orquestra. O `HotspotSQLAlchemyRepository` lida exclusivamente com a persistência 
(PostgreSQL), e o `hotspots.py` lida com o transporte HTTP (FastAPI) e o mapeamento de 
exceções de negócio (`HotspotAlreadyExistsError`) para o HTTP Status 409 (Conflict), 
conforme as boas práticas REST para Command. 
2.  **Tratamento de Unicidade (`IntegrityError`)**: Em vez de fazer uma consulta de checagem 
(`get_by_url`) antes do `create`, o Repositório se baseia na exceção de integridade do banco 
de dados (`IntegrityError`). Isso previne *race conditions* em ambientes concorrentes, 
garantindo que a unicidade seja imposta atomicamente pelo PostgreSQL, antes de mapearmos 
o erro para a exceção de negócio `HotspotAlreadyExistsError`. 
3.  [cite_start]**Segurança (Autenticação)**: O uso da dependência 
`Depends(get_current_user_mock)` garante que ambos os endpoints (POST e GET) exigem 
um mecanismo de autenticação (JWT)[cite: 112]. O mock simplifica o teste unitário, mas na 
produção, essa dependência seria implementada com a verificação real do token. 
4.  [cite_start]**Performance (Async/SQLAlchemy)**: O uso de `async/await` em todas as 
camadas (API, Service, Repository) com `AsyncSession` do SQLAlchemy (implícito no mock) 
garante que as operações de I/O sejam não-bloqueantes, maximizando o *throughput* do 
FastAPI, em linha com os requisitos de performance[cite: 274, 100]. 
5.  [cite_start]**Extensibilidade (Repository Pattern)**: A interface `HotspotRepository` permite, 
futuramente, trocar a implementação do PostgreSQL por MongoDB ou até mesmo adicionar um 
`HotspotRedisCacheRepository` entre o Service e o SQL Repository sem alterar a lógica de 
negócio do `HotspotService`[cite: 230]. [cite_start]A listagem (Query) é um candidato ideal para 
cache com Redis[cite: 187]. 
</design_notes>

        return self

from dataclasses import dataclass 
import math 
 
@dataclass(frozen=True) 
class PesosPontuacao: 
    """ 
    Value Object que representa os pesos de cada critério de pontuação. 
 
    Imutável, define a importância relativa de cada dimensão. 
    """ 
    peso_interessante: float 
    peso_bonita: float 
    peso_solteira: float 
 
    def __post_init__(self): 
        """Valida que a soma dos pesos é 1.0 (com tolerância).""" 
        soma = self.peso_interessante + self.peso_bonita + self.peso_solteira 
        # Usando math.isclose para robustez contra erros de ponto flutuante [cite: 1422] 
        if not math.isclose(soma, 1.0, abs_tol=0.01): 
            raise ValueError(f"Soma dos pesos deve ser 1.0 (recebido: {soma}).") 
 
@dataclass(frozen=True) 
class Pontuacao: 
    """ 
    Value Object que representa as pontuações brutas de um perfil. 
 
    Imutável, contém os scores de 0 a 10 de cada dimensão. 
    """ 



    interessante: float 
    bonita: float 
    solteira: float 
 
    def __post_init__(self): 
        """Valida que cada pontuação está no range [0, 10][cite: 1401].""" 
        fields = {'interessante': self.interessante, 'bonita': self.bonita, 'solteira': self.solteira} 
        for campo, valor in fields.items(): 
            if not (0.0 <= valor <= 10.0): 
                raise ValueError(f"A pontuação '{campo}' deve estar entre 0.0 e 10.0 (recebido: 
{valor}).") 
 
    def calcular_final(self, pesos: PesosPontuacao) -> float: 
        """ 
        Calcula a pontuação final como a média ponderada das pontuações brutas. 
         
        Args: 
            pesos: O Value Object PesosPontuacao com os pesos configurados. 
             
        Returns: 
            A pontuação final de 0 a 10. 
        """ 
        # Fórmula: (Score1 * Peso1 + Score2 * Peso2 + Score3 * Peso3) / Soma dos Pesos (que 
é 1.0) 
        # O resultado final é a média ponderada [cite: 1404] 
        pontuacao_final = ( 
            (self.interessante * pesos.peso_interessante) + 
            (self.bonita * pesos.peso_bonita) + 
            (self.solteira * pesos.peso_solteira) 
        ) 
        return round(pontuacao_final, 2)

    """Exceção customizada para falhas no cálculo de score (ex: dados incompletos).""" 
    pass 
 



class StrategyPontuacao(ABC): 
    """ 
    Interface (Abstract Strategy) para calcular a pontuação de uma dimensão do perfil. 
     
    Implementa o Strategy Pattern para injetar diferentes lógicas de score. 
    """ 
    @abstractmethod 
    def calcular(self, raw_analysis_data: Dict[str, Any]) -> float: 
        """ 
        Calcula a pontuação (0-10) para o critério específico. 
         
        Args: 
            raw_analysis_data: O dossiê completo (JSON) gerado pelo Agente Analista. 
             
        Returns: 
            O score de 0.0 a 10.0. 
         
        Raises: 
            ScoreCalculationError: Se dados cruciais estiverem faltando. 
        """ 
        pass 
 
class StrategyInteressante(StrategyPontuacao): 
    """Estratégia para calcular a pontuação de 'Interessante' (Compatibilidade de Interesses).""" 
    def calcular(self, raw_analysis_data: Dict[str, Any]) -> float: 
        """ 
        Simula o cálculo de score baseado em NLP (extração de interesses, hobbies, 
personalidade). 
         
        Returns: 
            Score de 0.0 a 10.0. 
        """ 
        try: 
            # Dado simulado que viria do NLP (Análise Textual) 
            match_score = raw_analysis_data['textual_analysis']['interest_match_pct']  
            # Conversão simples: 0% match = 0 score, 100% match = 10 score 
            score = match_score * 10.0 
            return round(min(score, 10.0), 2) 
        except KeyError: 
            # Caso os dados do NLP não estejam disponíveis 
            raise ScoreCalculationError("Dados de Análise Textual incompletos para score 
Interessante.") 
 
class StrategyBonita(StrategyPontuacao): 



    """Estratégia para calcular a pontuação de 'Bonita' (Atributos Estéticos).""" 
    def calcular(self, raw_analysis_data: Dict[str, Any]) -> float: 
        """ 
        Simula o cálculo de score baseado em Computer Vision (análise de fotos). 
         
        Returns: 
            Score de 0.0 a 10.0. 
        """ 
        try: 
            # Dado simulado que viria do Computer Vision (Análise Visual) 
            aesthetic_score = raw_analysis_data['visual_analysis']['aesthetic_rating'] 
            # O score estético (0-1) é normalizado para 0-10 
            score = aesthetic_score * 10.0  
            return round(min(score, 10.0), 2) 
        except KeyError: 
            raise ScoreCalculationError("Dados de Análise Visual incompletos para score Bonita.") 
 
class StrategySolteira(StrategyPontuacao): 
    """Estratégia para calcular a pontuação de 'Solteira' (Probabilidade de Estar Solteira).""" 
    def calcular(self, raw_analysis_data: Dict[str, Any]) -> float: 
        """ 
        Simula o cálculo de score baseado em NLP e Visão (inferência de status de 
relacionamento). 
         
        Returns: 
            Score de 0.0 a 10.0. 
        """ 
        try: 
            # Dado simulado que viria do inferência de ML 
            single_prob = raw_analysis_data['social_analysis']['single_probability'] 
            # Conversão simples: 0.8 probabilidade = 8.0 score 
            score = single_prob * 10.0 
            return round(min(score, 10.0), 2) 
        except KeyError: 
            raise ScoreCalculationError("Dados de Análise Social incompletos para score Solteira.")

# Imports internos: Perfil, PesosPontuacao, Pontuacao, StrategyPontuacao, 
ScoreCalculationError 



 
class PerfilScoreService: 
    """ 
    Serviço de Aplicação responsável por orquestrar o cálculo da pontuação de um Perfil. 
     
    Este serviço é acionado pelo Agente Analista. 
    """ 
    def __init__( 
        self, 
        strategies: Dict[str, Type[StrategyPontuacao]], 
        limiar_qualificacao: float = 7.5 
    ): 
        """ 
        Inicializa o serviço com as estratégias e o limiar de qualificação. 
 
        Args: 
            strategies: Mapeamento de nome do critério para a classe Strategy. 
            limiar_qualificacao: Pontuação mínima para o status 'QUALIFICADO'. 
        """ 
        self._strategies = strategies 
        self._limiar = limiar_qualificacao 
 
    async def calculate_and_qualify( 
        self, 
        perfil: Perfil, 
        raw_analysis_data: Dict[str, Any], 
        pesos: PesosPontuacao 
    ) -> Perfil: 
        """ 
        Executa todas as estratégias de score, calcula a pontuação final e atualiza o Perfil. 
         
        Args: 
            perfil: A entidade Perfil a ser atualizada. 
            raw_analysis_data: O dossiê de análise (dados brutos de ML/NLP). 
            pesos: Os pesos de configuração do usuário (VO). 
             
        Returns: 
            A entidade Perfil atualizada. 
             
        Raises: 
            ScoreCalculationError: Se alguma estratégia falhar devido a dados insuficientes. 
        """ 
         
        scores = {} 



        # 1. Executa cada estratégia 
        for criterion, strategy_class in self._strategies.items(): 
            strategy_instance = strategy_class() 
            try: 
                score = strategy_instance.calcular(raw_analysis_data) 
                scores[criterion] = score 
            except ScoreCalculationError as e: 
                # Logar a falha da estratégia sem interromper o pipeline principal (simplicidade) 
                # Em um sistema real, poderíamos lançar a exceção ou atribuir um score baixo. 
                # Aqui, lançaremos para garantir a integridade dos dados. 
                raise e 
 
        # 2. Cria o Value Object Pontuacao 
        pontuacao_vo = Pontuacao( 
            interessante=scores['interessante'], 
            bonita=scores['bonita'], 
            solteira=scores['solteira'] 
        ) 
 
        # 3. Calcula a pontuação final ponderada 
        pontuacao_final = pontuacao_vo.calcular_final(pesos) 
 
        # 4. Atualiza a entidade Perfil (lógica de estado no Aggregate Root) [cite: 1367] 
        perfil.qualificar(pontuacao_final, limiar=self._limiar) 
 
        return perfil

    'solteira': StrategySolteira,

<design_notes> 
1.  **Strategy Pattern (SRP e Desacoplamento)**: A implementação do Strategy Pattern com 
`StrategyPontuacao` atende diretamente ao requisito de manutenibilidade e 
extensibilidade[cite: 1707]. Cada critério de pontuação (`Interessante`, `Bonita`, `Solteira`) 
possui sua lógica isolada. Se a tecnologia para pontuar 'Bonita' mudar (ex: trocar de OpenCV 
para um modelo customizado), apenas a classe `StrategyBonita` precisa ser alterada, sem 
tocar no `PerfilScoreService` ou em outros critérios (SRP)[cite: 842]. 
2.  **Imutabilidade dos Value Objects (DDD)**: Os VOs `Pontuacao` e `PesosPontuacao` 
utilizam `dataclass(frozen=True)`[cite: 1392]. Isso garante que, uma vez criados, os dados de 
configuração (`PesosPontuacao`) e os dados de score (`Pontuacao`) não possam ser alterados 
acidentalmente, protegendo a integridade do cálculo. 
3.  **Lógica na Entidade (`Perfil.qualificar`)**: A decisão de mudar o status do perfil 
(`QUALIFICADO` ou `DESQUALIFICADO`) com base na pontuação (`pontuacao_final >= 
limiar`) está encapsulada dentro da Entidade `Perfil`[cite: 1367]. No DDD, o Aggregate Root 
(`Perfil`) deve conter a lógica de negócio que governa seu estado e suas transições, garantindo 
que o objeto esteja sempre em um estado consistente. 
4.  **Orquestração no Service Layer**: O `PerfilScoreService` atua como o **Context** no 
Strategy Pattern, orquestrando as execuções das estratégias e delegando o cálculo ponderado 
final para o `Value Object Pontuacao`. Ele é a ponte entre os dados brutos de análise (dossiê) e 
a Entidade de Domínio (Perfil), mantendo a lógica de negócio principal fora da camada de API. 
</design_notes>

class ServicoHandoff: 
    """ 
    Serviço de Aplicação para preparar e notificar a transição da conversa para o humano. 
    """ 
    def __init__( 
        self, 
        data_repository: DataRepository, 
        publisher: EventPublisher, 
    ): 
        """ 
        Args: 
            data_repository: Repositório unificado para leitura/escrita de dados de handoff. 
            publisher: Publicador de eventos. 
        """ 
        self._data_repo = data_repository 
        self._publisher = publisher 
 



    async def _simular_notificacao(self, pacote: PacoteContexto): 
        """ 
        Simula o envio de uma notificação crítica para o operador (ex: Telegram Bot API). 
         
        Raises: 
            NotificationError: Em caso de falha de comunicação com o serviço externo. 
        """ 
        if "simular_falha" in pacote.url_perfil: 
            raise NotificationError("Falha na API do Telegram") 
             
        print(f"[NOTIFICAÇÃO MOCK] Alerta! Handoff para Perfil: {pacote.url_perfil} (Score: 
{pacote.pontuacao_final:.1f})") 
        print(f"[NOTIFICAÇÃO MOCK] Próximo passo sugerido: {pacote.sugestao_abordagem}") 
 
 
    async def start_handoff(self, perfil_id: UUID, correlation_id: str) -> bool: 
        """ 
        Monta o Pacote de Contexto, notifica o operador e publica o evento HandoffIniciado. 
 
        Args: 
            perfil_id: ID do perfil que respondeu positivamente. 
            correlation_id: ID de rastreamento. 
 
        Returns: 
            True se o processo de handoff foi acionado. 
         
        Raises: 
            HandoffError: Se o perfil ou dados cruciais não forem encontrados. 
            NotificationError: Se a notificação falhar. 
        """ 
         
        # 1. Recuperação de Dados (Leitura de múltiplos DBs) 
        handoff_data = await self._data_repo.get_handoff_data(perfil_id) 
        if handoff_data is None: 
            raise HandoffError(f"Dados completos para Handoff não encontrados para Perfil ID: 
{perfil_id.hex[:4]}.") 
 
        # 2. Montagem do Pacote de Contexto (VO) 
        pacote = PacoteContexto.from_data(perfil_id, handoff_data) 
 
        # 3. Persistência do Pacote de Contexto (MongoDB) 
        # O dossiê completo é armazenado em DB NoSQL para consulta rápida do Dashboard. 
        pacote_id = await self._data_repo.persistir_pacote(pacote) 
         



        try: 
            # 4. Notificação do Operador Humano (I/O Externo) 
            await self._simular_notificacao(pacote) 
        except NotificationError as e: 
            # A falha na notificação não reverte a persistência do Pacote. 
            # Apenas logamos e prosseguimos para o evento (best effort para notificação). 
            print(f"[AVISO] Falha na notificação externa: {e}. Evento Handoff será publicado.") 
            # O erro de notificação é contido, mas é um ponto de falha. 
 
        # 5. Publicação do Evento (EDA) 
        await self._publisher.publicar_evento( 
            event_type="HandoffIniciado", 
            data={ 
                "pacote_id": str(pacote_id), # Apenas o ID do dossiê é publicado! (Segurança) 
                "perfil_id": str(perfil_id), 
            }, 
            correlation_id=correlation_id 
        ) 
 
        return True

<design_notes> 
1.  **Value Object (PacoteContexto) como Contrato**: O `PacoteContexto` é o contrato de 
transferência de conhecimento entre a IA/Automação e o Operador Humano. Ele encapsula o 
dossiê completo de forma imutável (`frozen=True`), garantindo que a informação consumida 
pelo humano seja coerente e consistente no momento do Handoff. A lógica de **sugestão de 
abordagem** é um pequeno trecho de lógica de negócio embutida no VO, aumentando seu 
valor no DDD. 



2.  **Segurança e Consistência (Handoff)**: A ordem das operações é crucial: 1. Recuperar 
dados; 2. Montar VO; 3. Persistir Pacote; 4. Notificar; 5. Publicar Evento. Ao publicar o evento 
`HandoffIniciado`, apenas o `pacote_id` é incluído. Isso garante que o dossiê completo (dados 
sensíveis) não trafegue pela fila de mensagens Redis (pública/não criptografada em trânsito), 
mantendo a segurança (ONF4). O próximo consumidor (Dashboard/Frontend) precisará buscar 
o pacote completo no MongoDB usando o ID. 
3.  **Dependência Cruzada (Múltiplos DBs)**: O `DataRepository` simula a necessidade de 
buscar dados em diferentes tecnologias (PostgreSQL para dados estruturados de `Perfil`, 
MongoDB para `Análise Completa`/`PacoteContexto`). Isso reflete o CQRS/Arquitetura de 
Microserviços onde a recuperação de dados complexos exige um *Data Access Layer* 
unificado. 
4.  **Resiliência na Notificação**: A notificação é um I/O externo passível de falha 
(`NotificationError`). A falha na notificação não reverte a persistência do `PacoteContexto` (que 
é o dado de negócio mais importante) nem impede a publicação do evento. O sistema segue o 
princípio do *best effort* para notificação, priorizando a continuidade do workflow. 
</design_notes>

class LLMGenerationError(Exception): 
    """Exceção customizada para falhas na geração de mensagens por LLM.""" 
    pass 
 
class LLMStrategy(ABC): 



    """ 
    Interface (Abstract Strategy) para geração de mensagens personalizadas. 
     
    Cada classe concreta implementa a lógica de chamada e custeio para um LLM específico. 
    """ 
    @abstractmethod 
    async def generate( 
        self, 
        estrategia_persuasao: str, 
        persona_config: Dict[str, Any], 
        perfil: Perfil, 
    ) -> MensagemGerada: 
        """ 
        Simula a chamada a API de LLM para gerar a mensagem. 
 
        Args: 
            estrategia_persuasao: O resumo da estratégia definida pelo Motor de Persuasão. 
            persona_config: Configuração da persona do Agente Escriba. 
            perfil: Entidade Perfil com todos os dados de análise. 
 
        Returns: 
            O Value Object MensagemGerada. 
 
        Raises: 
            LLMGenerationError: Se a geração falhar. 
        """ 
        pass 
 
class GPT4Strategy(LLMStrategy): 
    """Estratégia para usar o GPT-4 (Custo Alto, Qualidade Premium).""" 
    MODEL_NAME = "GPT-4" 
    COST_PER_MESSAGE = 0.005 # Simulado

        self,

        # A complexidade de montar o Prompt Mestre (que usa estrategia_persuasao, perfil, etc.)

        mensagem = (

# 3. application/agent/scribe_service.py (Service Layer)

class AgenteEscribaService: 
    """ 
    Serviço de orquestração do Agente Escriba. 
 
    Responsável por selecionar a melhor estratégia de LLM e gerar a mensagem. 
    """ 
    PREMIUM_SCORE_THRESHOLD = 8.5 
     
    def __init__(self, high_cost_strategy: Type[LLMStrategy], low_cost_strategy: 
Type[LLMStrategy]): 
        """ 
        Args: 
            high_cost_strategy: Estratégia premium (ex: GPT-4). 
            low_cost_strategy: Estratégia econômica (ex: GPT-4o-mini). 
        """ 
        self._high_cost_strategy = high_cost_strategy 
        self._low_cost_strategy = low_cost_strategy 
 
    def _select_strategy(self, pontuacao_final: float) -> LLMStrategy: 
        """ 
        Lógica de seleção do LLM baseada na pontuação final do perfil. 
         
        Args: 
            pontuacao_final: A pontuação final do perfil (0-10). 
 
        Returns: 
            Uma instância da LLMStrategy apropriada. 
        """ 
        # Se pontuação for premium (>= 8.5), justifica o custo mais alto do GPT-4 
        if pontuacao_final >= self.PREMIUM_SCORE_THRESHOLD: 
            # Seleção Strategy Pattern 
            return self._high_cost_strategy() 
        else: 
            # Caso contrário, usa a opção econômica 
            return self._low_cost_strategy() 
 
    async def generate_message( 
        self, 
        perfil: Perfil, 
        estrategia_persuasao: str, 
        persona_config: Dict[str, Any] 
    ) -> MensagemGerada: 
        """ 
        Orquestra a seleção do LLM e a geração da mensagem. 



 
        Args: 
            perfil: Entidade Perfil com pontuação. 
            estrategia_persuasao: Resumo da estratégia (ex: "Afeição, mencionando o show de 
rock"). 
            persona_config: Configurações da persona do Agente Escriba. 
 
        Returns: 
            O Value Object MensagemGerada. 
 
        Raises: 
            LLMGenerationError: Se o perfil não tiver pontuação final. 
        """ 
        if perfil.pontuacao_final is None: 
            raise LLMGenerationError(f"Perfil {perfil.id} não possui pontuação final. Análise 
incompleta.") 
         
        # 1. Seleção da Estratégia (Strategy Pattern) 
        selected_strategy = self._select_strategy(perfil.pontuacao_final) 
 
        # 2. Execução da Estratégia 
        mensagem_gerada = await selected_strategy.generate( 
            estrategia_persuasao=estrategia_persuasao, 
            persona_config=persona_config, 
            perfil=perfil 
        ) 
         
        return mensagem_gerada

<design_notes> 



1.  **Strategy Pattern para Otimização de Custo/Qualidade**: O uso do Strategy Pattern é ideal 
aqui. Ele resolve o problema de otimização de custo/qualidade [cite: 1325, 1370] ao delegar a 
lógica de custo e chamada de API a classes específicas (`GPT4Strategy`, 
`GPT4OMiniStrategy`). O `AgenteEscribaService` atua apenas como um seletor ($O(1)$), 
garantindo que perfis altamente qualificados (`>= 8.5`) recebam a abordagem premium, 
justificando o custo mais alto. 
2.  **Imutabilidade e Encapsulamento de Saída**: O `MensagemGerada` como Value Object 
(`frozen=True`) garante que o resultado da geração e seu custo associado sejam encapsulados 
de forma imutável. Isso é crucial para persistência e auditoria de custos no log. 
3.  **Separação de Preocupações (SRP)**: A lógica de **seleção** de LLM reside no 
`AgenteEscribaService` (Application Layer/Business Rule), enquanto a lógica de **interação** 
com a API (montagem de prompt, chamada, custeio) reside nas classes `LLMStrategy` 
(Infrastructure/Domain). Essa separação facilita a troca e a manutenção das integrações com 
as APIs de IA[cite: 149]. 
4.  **Tratamento de Estado**: O serviço verifica se o perfil tem `pontuacao_final` [cite: 523] 
antes de prosseguir. Isso impede que o Agente Escriba tente processar perfis que não 
passaram pelo Agente Analista (violação do fluxo de eventos). 
</design_notes>

class AgenteEscribaService: 
    """ 
    Serviço de orquestração do Agente Escriba (Refatorado para Gemini). 
    """ 
    PREMIUM_SCORE_THRESHOLD = 8.5 
     
    def __init__(self, high_cost_strategy: Type[LLMStrategy], low_cost_strategy: 
Type[LLMStrategy]): 
        """ 
        Args: 
            high_cost_strategy: Estratégia premium (ex: Gemini Ultra). 
            low_cost_strategy: Estratégia econômica (ex: Gemini Flash). 



        """ 
        self._high_cost_strategy = high_cost_strategy 
        self._low_cost_strategy = low_cost_strategy 
 
    def _select_strategy(self, pontuacao_final: float) -> LLMStrategy: 
        """ 
        Lógica de seleção do LLM baseada na pontuação final do perfil. 
         
        A lógica de seleção permanece inalterada, demonstrando a robustez do Strategy Pattern. 
        """ 
        # Se pontuação for premium (>= 8.5), usa a estratégia de custo alto. 
        if pontuacao_final >= self.PREMIUM_SCORE_THRESHOLD: 
            return self._high_cost_strategy() 
        else: 
            # Caso contrário, usa a opção econômica. 
            return self._low_cost_strategy() 
 
    async def generate_message( 
        self, 
        perfil: Perfil, 
        estrategia_persuasao: str, 
        persona_config: Dict[str, Any] 
    ) -> MensagemGerada: 
        """ 
        Orquestra a seleção do LLM e a geração da mensagem. 
        """ 
        if perfil.pontuacao_final is None: 
            raise LLMGenerationError(f"Perfil {perfil.id} não possui pontuação final. Análise 
incompleta.") 
         
        # 1. Seleção da Estratégia (Strategy Pattern) 
        selected_strategy = self._select_strategy(perfil.pontuacao_final) 
 
        # 2. Execução da Estratégia 
        mensagem_gerada = await selected_strategy.generate( 
            estrategia_persuasao=estrategia_persuasao, 
            persona_config=persona_config, 
            perfil=perfil 
        ) 
         
        return mensagem_gerada

se a estratégia é *high cost* ou *low cost*, o nome exato do modelo e sua implementação de 
API estão isolados nas classes `LLMStrategy` concretas. 
2.  **Manutenção de Custo e Qualidade**: O limiar de `8.5` e os custos simulados (`0.005` e 
`0.0001`) foram mantidos, garantindo que a regra de negócio de otimização de custos continue 
válida, agora usando a nova stack de IA (Gemini). 
3.  **Aderência ao DDD**: O Value Object `MensagemGerada` e a Entidade `Perfil` foram 
mantidos intactos, provando que o domínio de negócio é robusto e independente da tecnologia 
de infraestrutura (a API de LLM).

class AgenteAprendizagemService: 
    """ 
    Serviço de Orquestração do Agente de Aprendizagem. 
 
    Responsável por converter o feedback humano em uma Recompensa de ML e persistir o 
episódio. 
    """ 
    def __init__( 
        self, 
        repository: DataRepository, 
        publisher: EventPublisher, 
    ): 
        self._repository = repository 
        self._publisher = publisher 
        # Define os pesos base para o cálculo da recompensa 
        self.WEIGHT_SUCCESS = 0.5 
        self.WEIGHT_FAILURE = 0.0 
        self.WEIGHT_BLOCK = -0.2 
        self.WEIGHT_RATING_BASE = 0.5 # A avaliação humana contribui com 50% do score 
final 
 
    def _calculate_reward(self, feedback: FeedbackReceived) -> float: 
        """ 
        Calcula a recompensa final (float 0.0 a 1.0) para o modelo de Aprendizado por Reforço. 



 
        Args: 
            feedback: O Value Object FeedbackReceived. 
 
        Returns: 
            A recompensa final. 
        """ 
        # 1. Componente de Avaliação Humana (Base: 0.0 a 1.0) 
        # Média das avaliações: (1 a 5) -> (0.2 a 1.0) 
        avg_rating = (feedback.avaliacao_perfil + feedback.avaliacao_mensagem) / 2 
         
        # Normaliza a avaliação para um peso de 0.0 a 0.5 
        rating_reward = (avg_rating / 5.0) * self.WEIGHT_RATING_BASE 
         
        # 2. Componente de Resultado de Negócio (Base: SUCESSO/FALHA/BLOQUEIO) 
        if feedback.resultado_final == ResultadoFinalEnum.SUCESSO: 
            business_reward = self.WEIGHT_SUCCESS 
        elif feedback.resultado_final == ResultadoFinalEnum.BLOQUEIO: 
            business_reward = self.WEIGHT_BLOCK 
        else: # FALHA 
            business_reward = self.WEIGHT_FAILURE 
             
        # 3. Recompensa Final: Soma dos componentes 
        final_reward = round(rating_reward + business_reward, 4) 
         
        # Garante que o resultado é razoável (após arredondamento) 
        return max(0.0, min(1.0, final_reward)) 
 
    async def process_feedback(self, feedback: FeedbackReceived, correlation_id: str) -> UUID: 
        """ 
        Processa o feedback, calcula a recompensa, persiste e publica o evento de retreinamento. 
 
        Args: 
            feedback: O feedback recebido (Command). 
            correlation_id: ID de rastreamento. 
 
        Returns: 
            O ID do Episódio de Treinamento criado. 
         
        Raises: 
            LearningError: Se a persistência falhar. 
        """ 
         
        # 1. Cálculo da Recompensa (Lógica de Negócio de MLOps) 



        recompensa = self._calculate_reward(feedback) 
         
        # 2. Criação da Entidade 
        episodio = EpisodioTreinamento( 
            pacote_id=feedback.pacote_id, 
            recompensa_final=recompensa, 
            resultado=feedback.resultado_final 
        ) 
         
        # 3. Persistência do Episódio 
        episodio_id = await self._repository.persistir_episodio(episodio) 
 
        # 4. Publicação do Evento (Gatilho para o pipeline MLOps) 
        await self._publisher.publicar_evento( 
            event_type="FeedbackRegistrado", 
            data={ 
                "episodio_id": str(episodio_id), 
                "recompensa": recompensa, 
                "resultado": episodio.resultado.value 
            }, 
            correlation_id=correlation_id 
        ) 
 
        return episodio_id

<design_notes> 
1.  **Fechamento do Loop de MLOps (RLHF)**: Esta tarefa finaliza o pipeline do Projeto 
Quimera, ligando o resultado da interação humana (Feedback) de volta ao Agente de 
Aprendizagem. O `AgenteAprendizagemService` é o coração dessa lógica, convertendo o 
feedback subjetivo em uma **recompensa quantitativa** para o modelo de ML. 
2.  **Entidade `EpisodioTreinamento` (DDD)**: O `EpisodioTreinamento` atua como o agregado 
principal para a camada de aprendizado. Ele armazena o estado anterior (via `pacote_id`) e a 
recompensa calculada, que é o dado essencial para o algoritmo de Retreinamento por Reforço 
(RLHF). 
3.  **Command Pattern e Performance**: O endpoint `POST /learning/feedback` é um 
`Command`, garantindo que o comando seja processado de forma transacional. O cálculo da 
recompensa é instantâneo, e o I/O é limitado à persistência e à publicação de um evento 
simples, garantindo a baixa latência ($<50ms$). 
4.  **Segurança e Consistência**: O `FeedbackReceived` usa Pydantic (`conint(ge=1, le=5)`) 
para validação rigorosa dos inputs, prevenindo que dados inconsistentes cheguem à lógica de 



cálculo de recompensa. O JWT mockado garante que apenas operadores autenticados possam 
enviar feedback. 
5.  **Gatilho EDA**: A publicação do evento `FeedbackRegistrado` é o gatilho final para o 
pipeline assíncrono de MLOps. Este evento seria consumido pelo microsserviço de treinamento 
(fora do escopo), que periodicamente coleta todos os novos episódios para retreinar o modelo 
de score e estratégia. 
</design_notes>

class SentimentStrategy(abc.ABC): 
    """ 
    Interface (Abstract Strategy) para a classificação de sentimento. 
    """ 
    @abc.abstractmethod 
    def classify(self, text: str) -> SentimentoEnum: 
        """ 
        Classifica o texto como Positivo, Neutro ou Negativo. 
        """ 
        pass 
 
class PositiveSentimentStrategy(SentimentStrategy): 
    """ 
    Estratégia Positiva: Busca por alto engajamento. (Prioritário para Handoff) 
    """ 
    POSITIVE_KEYWORDS = ["adorei", "ótima ideia", "claro", "sim!", "quero", "legal", "massa", 
"interessante"] 
     
    def classify(self, text: str) -> SentimentoEnum: 
        """ 
        Classifica como POSITIVO se encontrar palavras-chave de alto engajamento. 
        """ 
        lower_text = text.lower() 
        if any(keyword in lower_text for keyword in self.POSITIVE_KEYWORDS): 
            return SentimentoEnum.POSITIVO 
        # Se não for Positivo, a estratégia Neutra/Negativa deve ser consultada 
        return SentimentoEnum.NEUTRO # Retorno intermediário 
 
class NeutralNegativeStrategy(SentimentStrategy): 
    """ 
    Estratégia Neutra/Negativa: Classificação padrão para baixo engajamento ou rejeição. 
    """ 



    NEGATIVE_KEYWORDS = ["não tenho interesse", "pare", "rude", "bloquear", "quem é você", 
"não lembro"] 
 
    def classify(self, text: str) -> SentimentoEnum: 
        """ 
        Classifica como NEGATIVO se encontrar palavras de rejeição, senão NEUTRO. 
        """ 
        lower_text = text.lower() 
        if any(keyword in lower_text for keyword in self.NEGATIVE_KEYWORDS): 
            return SentimentoEnum.NEGATIVO 
        return SentimentoEnum.NEUTRO

class AgenteMonitoramentoService: 
    """ 
    Serviço de Orquestração do Agente de Monitoramento. 
     
    Responsável por: 1. Polling; 2. Classificação de Sentimento; 3. Handoff Decision. 
    """ 
    HANDOFF_LIMIAR = SentimentoEnum.POSITIVO 
     
    def __init__( 
        self, 
        repository: PerfilRepository, 
        publisher: EventPublisher, 
        positive_strategy: SentimentStrategy, 
        negative_strategy: SentimentStrategy 
    ): 
        """ 
        Args: 
            repository: Repositório para atualizar o status do perfil. 
            publisher: Publicador de eventos. 
            positive_strategy: Estratégia para detecção de Positivo. 
            negative_strategy: Estratégia para detecção de Neutro/Negativo. 
        """ 
        self._repository = repository 
        self._publisher = publisher 
        self._positive_strategy = positive_strategy 
        self._negative_strategy = negative_strategy 
 



    def _classify_sentiment(self, text: str) -> SentimentoEnum: 
        """ 
        Usa o Strategy Pattern para classificar o sentimento. 
        """ 
        # 1. Tenta a classificação Positiva 
        result = self._positive_strategy.classify(text) 
         
        # 2. Se não for POSITIVO, tenta a classificação Neutra/Negativa 
        if result != SentimentoEnum.POSITIVO: 
            return self._negative_strategy.classify(text) 
             
        return result 
 
    async def _simular_scan_inbox(self) -> List[Dict[str, Any]]: 
        """ 
        Simula a busca por novas respostas na caixa de entrada (polling). 
         
        Em um sistema real, faria web scraping com Selenium/Playwright a cada 15 minutos. 
        """ 
        # Retorna uma lista de perfis que "responderam" 
        return [ 
            {"perfil_id": str(uuid4()), "texto": "Oi, adorei a ideia! Você é de Varginha?"}, 
            {"perfil_id": str(uuid4()), "texto": "Obrigado, mas não tenho interesse no momento."}, 
            {"perfil_id": str(uuid4()), "texto": "ok"}, 
        ] 
 
 
    async def execute_monitoring( 
        self, 
        perfil: Perfil, 
        resposta_texto: str, 
        correlation_id: str 
    ) -> bool: 
        """ 
        Executa a verificação, classificação e decisão de handoff para uma resposta. 
 
        Args: 
            perfil: Perfil que respondeu (com status 'ENVIADA'). 
            resposta_texto: O texto da mensagem de resposta. 
            correlation_id: ID de rastreamento. 
 
        Returns: 
            True se o Handoff for acionado. 
        """ 



         
        # 1. Validação de Pré-condição 
        if perfil.status != ProfileStatusEnum.ENVIADA: 
            raise MonitoringError(f"Perfil {perfil.id.hex[:4]} não está ENVIADA. Status: 
{perfil.status}.") 
         
        # 2. Classificação de Sentimento (Strategy) 
        sentimento = self._classify_sentiment(resposta_texto) 
         
        resposta_vo = RespostaDetectada( 
            perfil_id=perfil.id, 
            texto_resposta=resposta_texto, 
            sentimento=sentimento 
        ) 
         
        # 3. Decisão de Handoff 
        if sentimento == self.HANDOFF_LIMIAR: 
            # 3.1. Persistência: Atualiza o status para RESPONDEU 
            updated_perfil = await self._repository.update_status(perfil, 
ProfileStatusEnum.RESPONDEU) 
 
            # 3.2. Publicação do Evento: Aciona o Serviço de Handoff 
            await self._publisher.publicar_evento( 
                event_type="RespostaDetectada", 
                data={ 
                    "perfil_id": str(updated_perfil.id), 
                    "sentimento": sentimento.value, 
                    "texto_resposta": resposta_vo.texto_resposta, 
                }, 
                correlation_id=correlation_id 
            ) 
            return True 
        else: 
            # Não aciona handoff para neutro/negativo 
            return False

<design_notes> 
1.  **Handoff como Ponto de Decisão (Fase 7/8)**: O `AgenteMonitoramentoService` é o 
componente que conecta o mundo automatizado ao mundo humano (Handoff). Ele garante que 
o Handoff (publicação do evento `RespostaDetectada`) ocorra **somente** quando o 
sentimento for `POSITIVO`, que é o `HANDOFF_LIMIAR`. Isso filtra a carga de trabalho do 
usuário humano. 
2.  **Strategy Pattern para Sentimento (Extensibilidade)**: A classificação de sentimento usa o 
Strategy Pattern. A lógica atual é baseada em palavras-chave simples. Isso permite que, no 
futuro, a `PositiveSentimentStrategy` seja substituída por uma `GeminiFlashSentimentStrategy` 
(usando o LLM para classificação zero-shot) sem alterar a orquestração 
(`AgenteMonitoramentoService`). 
3.  **Transação Lógica (Persistência-Comunicação)**: A ordem é crucial: Classificação -> 
Update DB (`RESPONDEU`) -> Publish Event. Se a atualização do DB falhar 
(`ProfileUpdateError`), o evento **não é publicado**, evitando que o próximo serviço (Handoff) 
tente agir em um perfil cujo estado não foi persistido, mantendo a consistência lógica. 
4.  **Aderência ao EDA**: A publicação do evento `RespostaDetectada` é o gatilho assíncrono 
para o **Serviço de Handoff**, que é o próximo microserviço na arquitetura, conforme o fluxo de 
eventos. 
</design_notes>

import redis.asyncio as redis 
import json 
from uuid import uuid4 
from datetime import datetime 
 
class EventPublishError(Exception): 
    """Exceção lançada quando a publicação de um evento falha no Redis.""" 
    pass 
 
class EventPublisher: 
    """ 
    Publicador de Eventos que utiliza Redis Pub/Sub para comunicação assíncrona. 
 
    Publica eventos no canal específico do tipo de evento e em um canal geral 
    para auditoria. 
    """ 
    def __init__(self, redis_client: redis.Redis): 
        """ 
        Args: 
            redis_client: Uma instância do cliente Redis assíncrono. 
        """ 
        self._redis = redis_client 



        self._source = "agente_analista" 
 
    async def publicar_evento( 
        self, 
        event_type: str, 
        data: Dict[str, Any], 
        correlation_id: Optional[str] = None 
    ): 
        """ 
        Publica um evento JSON no canal Redis. 
 
        Args: 
            event_type: O nome do evento (ex: PerfilAnalisado). 
            data: O payload específico do evento. 
            correlation_id: ID para rastreamento distribuído (Tracing). 
 
        Raises: 
            EventPublishError: Se a operação de publicação falhar. 
        """ 
        evento = { 
            "event_type": event_type, 
            "event_id": str(uuid4()), 
            "timestamp": datetime.utcnow().isoformat() + "Z", 
            "source": self._source, 
            "data": data, 
            "metadata": { 
                "correlation_id": correlation_id or str(uuid4()), 
                # Outros metadados de rastreamento podem ser incluídos aqui 
            } 
        } 
         
        canal_especifico = f"eventos:{event_type}" 
        canal_auditoria = "eventos:todos" 
 
        try: 
            # Publicação no canal específico para o Agente Analista 
            await self._redis.publish(canal_especifico, json.dumps(evento)) 
            # Publicação no canal geral para auditoria (melhor esforço) 
            await self._redis.publish(canal_auditoria, json.dumps(evento)) 
             
            # Não retornar o evento serializado no pub/sub é a prática usual 
        except Exception as e: 
            raise EventPublishError(f"Falha ao publicar evento {event_type} no Redis: {e}")

    async def update_score_and_status(self, perfil: Perfil) -> Perfil: 
        """ 
        Atualiza pontuação, status e timestamp de análise do perfil de forma atômica. 
 
        Args: 
            perfil: A entidade Perfil com os campos já atualizados. 
 
        Returns: 
            A entidade Perfil atualizada. 
 
        Raises: 
            ProfileUpdateError: Em caso de falha de persistência. 
        """ 



        try: 
            # Mapeamento da Entidade para a instrução SQL 
            stmt = ( 
                update(PerfilModel) 
                .where(PerfilModel.id == perfil.id) 
                .values( 
                    status=perfil.status, 
                    pontuacao_final=perfil.pontuacao_final, 
                    timestamp_analise=perfil.timestamp_analise, 
                    data_atualizacao=func.now() # Atualiza o timestamp 
                ) 
            ) 
            await self._session.execute(stmt) 
            await self._session.commit() 
             
            # Recarregar/Atualizar a entidade local com o timestamp do DB seria o ideal, 
            # mas simplificamos aceitando a entidade passada. 
            perfil.data_atualizacao = datetime.utcnow() 
            return perfil 
        except SQLAlchemyError as e: 
            await self._session.rollback() 
            raise ProfileUpdateError(f"Falha ao atualizar score/status do perfil {perfil.id}: {e}")

class AgenteAnalistaService:

    Serviço de Orquestração do Agente Analista. 
 
    Responsável por: 1. Calcular Score; 2. Persistir Resultado; 3. Publicar Evento. 
    """

        self, 
        score_service: PerfilScoreService, 
        repository: PerfilRepository, 
        publisher: EventPublisher,

        Args: 
            score_service: O serviço de cálculo de score (lógica de negócio). 



            repository: O repositório de persistência. 
            publisher: O publicador de eventos assíncronos. 
        """

        self._repository = repository 
        self._publisher = publisher 
 
    async def execute_analysis( 
        self, 
        perfil: Perfil, 
        raw_analysis_data: Dict[str, Any], 
        pesos: PesosPontuacao, 
        correlation_id: str 
    ) -> bool:

        Executa a análise completa, persiste e notifica o próximo agente. 
 
        Args: 
            perfil: O perfil a ser analisado. 
            raw_analysis_data: Dados brutos de ML/NLP. 
            pesos: Pesos de pontuação do usuário. 
            correlation_id: ID de rastreamento da requisição. 
 
        Returns: 
            True se a transação for bem-sucedida. 
 
        Raises: 
            Exception: Em caso de qualquer falha no cálculo, persistência ou publicação. 
        """ 
         
        # 1. Lógica de Negócio: Calcular Score e Qualificar 
        perfil_qualificado = await self._score_service.calculate_and_qualify( 
            perfil, raw_analysis_data, pesos 
        )

            event_type="PerfilAnalisado",

                "perfil_id": str(updated_perfil.id), 



                "pontuacao_final": updated_perfil.pontuacao_final, 
                "status": updated_perfil.status, 
                # Não incluir dados sensíveis (PII) aqui [cite: 884] 
            }, 
            correlation_id=correlation_id 
        ) 
 
        return True

<design_notes> 
1.  **Orquestração no `AgenteAnalistaService`**: O Serviço de Aplicação funciona como o 
orquestrador do Agente Analista. Ele garante a ordem das operações (Lógica -> Persistência -> 
Comunicação) e o tratamento de exceções. A falha na Persistência (`ProfileUpdateError`) evita 
que a etapa de Comunicação (`publisher.publicar_evento`) seja executada, respeitando a 
garantia de que o evento só é publicado se a atualização do DB for bem-sucedida (o "Best 
Effort" para Transação Distribuída, aceitando consistência eventual [cite: 935]). 
2.  **EDA e Desacoplamento**: O uso do `EventPublisher` injetado e do Redis Pub/Sub 
desacopla o Agente Analista do Motor de Persuasão[cite: 850, 955]. O Agente Analista não se 
importa com quem consome o evento `PerfilAnalisado`, apenas que ele seja publicado[cite: 
1101]. 
3.  **Implementação do Publisher**: O `EventPublisher` publica a mensagem em dois canais: o 
específico (`eventos:PerfilAnalisado`) para o próximo agente (Motor de Persuasão) e o geral 
(`eventos:todos`) para o Agente de Auditoria/Observabilidade (Requisito ONF6)[cite: 1029, 
1107]. 
4.  **Segurança (PII)**: O payload do evento `PerfilAnalisado` contém apenas IDs, status e 
pontuação. URLs, biografias ou outros dados de PII (Personally Identifiable Information) não 
são incluídos no evento, pois não são necessários para o próximo agente, mitigando o risco de 
vazamento de dados sensíveis em filas de mensagens[cite: 884]. 
5.  **Performance de I/O**: A persistência usa o padrão de `update` do SQLAlchemy (SQL 
Alchemy Core/ORM), que é mais performático que a recuperação e salvamento do objeto 
completo. A comunicação utiliza o cliente Redis assíncrono para garantir baixa latência[cite: 
980]. 
</design_notes>

class MessageRateLimiter: 
    """ 
    Simulação simples do Token Bucket (para uso diário), implementado em memória. 
 
    Limite: 20 mensagens por dia por conta (simulada como ID único). 
    """ 
    MAX_DAILY_MESSAGES = 20 
    # { 'user_id': {'date': datetime.date(2025, 1, 1), 'count': 5} } 
    _counters: Dict[str, Dict[str, Any]] = {} 
 
    def __init__(self, account_id: str): 
        """ 
        Args: 
            account_id: O ID da conta de envio (isca) que está sendo usada. 
        """ 
        self._account_id = account_id 
        # Inicializa o contador se for a primeira vez 
        if self._account_id not in self._counters: 
            self._reset_counter() 
 
    def _reset_counter(self): 
        """Reseta o contador diário para o dia atual.""" 
        self._counters[self._account_id] = { 
            'date': date.today(), 
            'count': 0 
        } 



 
    def _get_count_and_check_reset(self) -> int: 
        """Verifica se a data mudou e retorna a contagem atualizada.""" 
        current_data = self._counters[self._account_id] 
        if current_data['date'] != date.today(): 
            self._reset_counter() 
            return 0 
        return current_data['count'] 
 
    def permitir_envio(self) -> bool: 
        """ 
        Verifica se o limite diário foi excedido. 
         
        Returns: 
            True se o envio for permitido, False caso contrário. 
        """ 
        current_count = self._get_count_and_check_reset() 
        return current_count < self.MAX_DAILY_MESSAGES 
 
    def incrementar_uso(self) -> None: 
        """ 
        Incrementa o contador de uso diário. 
         
        Raises: 
            ExecutionError: Se tentar incrementar após o limite. 
        """ 
        current_count = self._get_count_and_check_reset() 
        if current_count >= self.MAX_DAILY_MESSAGES: 
            raise ExecutionError("Tentativa de incrementar uso além do limite permitido.") 
         
        self._counters[self._account_id]['count'] += 1

class AgenteExecucaoService: 
    """ 
    Serviço de Orquestração do Agente de Execução (Envio de Mensagem). 
    """ 
    # Custo simulado de execução (energia, proxy, etc.) por mensagem 
    EXECUTION_COST = 0.001  
     
    def __init__( 



        self, 
        repository: PerfilRepository, 
        publisher: EventPublisher, 
        rate_limiter: MessageRateLimiter, # Injete a instância do Rate Limiter 
    ): 
        """ 
        Args: 
            repository: Repositório para atualizar o status do perfil. 
            publisher: Publicador de eventos. 
            rate_limiter: Instância do Rate Limiter para a conta de envio. 
        """ 
        self._repository = repository 
        self._publisher = publisher 
        self._limiter = rate_limiter 
 
    async def _simular_envio_furtivo(self) -> bool: 
        """ 
        Simula a digitação e o envio da mensagem com delays realistas. 
         
        Returns: 
            True se a simulação for bem-sucedida (simulada para 100% de sucesso). 
        """ 
        # Simulação de digitação (100-300ms por caractere - simulação simplificada) 
        delay = random.uniform(1.0, 3.0)  
        await asyncio.sleep(delay) 
         
        # Simulação de movimento de mouse/clique (o Agente Batedor faz o mais complexo) 
        await asyncio.sleep(random.uniform(0.5, 1.0)) 
 
        return True 
 
    async def execute_send( 
        self, 
        perfil: Perfil, 
        mensagem_vo: MensagemGerada, 
        account_id: str, 
        correlation_id: str 
    ) -> EnvioResultado: 
        """ 
        Executa o processo de envio completo, incluindo rate limiting e atualização de estado. 
 
        Args: 
            perfil: O perfil alvo. 
            mensagem_vo: O VO com a mensagem gerada e custo. 



            account_id: ID da conta de envio usada. 
            correlation_id: ID de rastreamento da requisição. 
 
        Returns: 
            O Value Object EnvioResultado. 
        """ 
        # 0. Validação de Pré-condição (DDD) 
        if perfil.status != ProfileStatusEnum.QUALIFICADO: 
            raise ExecutionError(f"Perfil {perfil.id.hex[:4]} não está QUALIFICADO. Status: 
{perfil.status}.") 
         
        # 1. Verificação de Rate Limit 
        if not self._limiter.permitir_envio(): 
            return EnvioResultado( 
                sucesso=False, 
                motivo_falha="RATE_LIMIT_EXCEEDED", 
                custo_total=0.0 
            ) 
 
        try: 
            # 2. Execução Furtiva 
            await self._simular_envio_furtivo() 
             
            # 3. Incrementa o uso após o envio bem-sucedido (importante para o limite) 
            self._limiter.incrementar_uso() 
 
            # 4. Atualização de Status (Persistência) 
            custo_total = mensagem_vo.custo_simulado + self.EXECUTION_COST 
             
            updated_perfil = await self._repository.update_status(perfil, 
ProfileStatusEnum.ENVIADA) 
 
            # 5. Publicação do Evento 
            await self._publisher.publicar_evento( 
                event_type="MensagemEnviada", 
                data={ 
                    "perfil_id": str(updated_perfil.id), 
                    "account_id": account_id, 
                    "custo_total": custo_total, 
                }, 
                correlation_id=correlation_id 
            ) 
             
            # 6. Retorno de Sucesso 



            return EnvioResultado( 
                sucesso=True, 
                motivo_falha=None, 
                custo_total=custo_total 
            ) 
 
        except ProfileUpdateError as e: 
            # Falha de DB após o envio (falha na transação) 
            raise ExecutionError(f"Falha na persistência após envio bem-sucedido: {e}") 
        except Exception as e: 
            # Outros erros (ex: falha na publicação do evento) 
            raise ExecutionError(f"Falha inesperada durante a execução: {e}")

<design_notes> 
1.  **Rate Limiter (Anti-Detecção)**: O `MessageRateLimiter` é a principal linha de defesa 
contra o banimento neste agente. A implementação simples, em memória, serve como um 
*Token Bucket* de limite diário. O design permite que, no futuro, ele seja facilmente refatorado 
para usar Redis para persistência de estado entre processos/instâncias do microserviço, sem 
alterar a interface `permitir_envio/incrementar_uso`. 
2.  **Lógica de Falha no VO de Saída**: O Rate Limiting é um cenário de falha **esperado** no 
domínio (evitar banimento). Por isso, o serviço retorna um `EnvioResultado` com 
`sucesso=False` e o `motivo_falha`, em vez de lançar uma exceção. Isso permite que o 
Orquestrador (não implementado aqui) trate essa falha de forma elegante (ex: agendar para o 
próximo dia) sem quebrar o workflow. 
3.  **Encapsulamento do Envio Furtivo**: A função `_simular_envio_furtivo` encapsula os 
delays assíncronos e randômicos[cite: 590, 592], cumprindo o requisito de simulação humana. 
O uso de `asyncio.sleep` garante que o thread do Agente não seja bloqueado, permitindo alto 
*throughput* de orquestração (se houvesse múltiplas tarefas de envio). 
4.  **Aderência ao EDA/Transação**: O incremento do uso, a atualização do DB (`ENVIADA`), 
e a publicação do evento (`MensagemEnviada`) ocorrem **somente** após a simulação de 
envio bem-sucedida e dentro de um bloco `try/except` que trata a falha de persistência 
(`ProfileUpdateError`) com uma `ExecutionError`, garantindo a ordem transacional lógica. 
5.  **Custo e Auditoria**: O `EnvioResultado` captura o `custo_total` (LLM + Execução), 
essencial para as métricas de negócio e auditoria de custos. 
</design_notes>
